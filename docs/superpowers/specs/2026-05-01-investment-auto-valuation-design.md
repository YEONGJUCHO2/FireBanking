## 1. 제품 결정

이 기능의 목표는 포트폴리오 관리가 아니다. 목표는 투자가능 자산 입력값을 더 검증 가능하게 만들어 **FIRE 예상일의 신뢰도**를 높이는 것이다.

제품 약속은 아래 문장으로 고정한다.

```text
국내 상장 보유 종목을 마지막 거래일 종가 기준으로 자동 평가하고,
그 값을 월말 FIRE 스냅샷에 고정한다.
```

이렇게 정의하면 Fire Banking은 주식 투자 앱이나 증권 계좌 통합 앱으로 흐르지 않고, 월 1회 부부 FIRE 체크인 앱이라는 정체성을 유지한다.

## 2. 1차 릴리즈 범위

포함한다.

- 국내주식.
- 국내 ETF.
- 국내상장 미국 노출 ETF. 예: TIGER/KODEX/ACE/SOL 미국 지수 ETF.
- 종목 검색 + 수량 입력 방식의 보유 종목 등록.
- 기타 투자자산 수동 입력.
- 연금저축/IRP 계좌 성격 저장.
- 자산에 연결되는 부채 잔액.
- 부채 이자와 원금상환의 월 현금흐름 반영.
- 월말 스냅샷 고정.
- 대시보드에서 과거 월별 스냅샷 조회.

포함하지 않는다.

- VOO, QQQ, SCHD, VTI, AAPL 같은 미국상장 주식/ETF 자동평가.
- USD/KRW 환율 자동 반영.
- 증권사 계좌 연동.
- CSV 가져오기.
- 붙여넣기 일괄 등록.
- 매도대금 현금성 자산 원장.
- 포트폴리오 비중/성과 차트.
- 투자 추천 또는 리밸런싱.
- AI 투자 분석.

미국상장 보유 종목은 이번 릴리즈에서 `기타 투자자산`으로 수동 입력한다.

1차 릴리즈는 하나의 제품 릴리즈로 묶되, 구현은 아래 내부 phase로 나눈다.

```text
Phase 0. Provider spike
- 키움 API로 국내 ETF와 국내상장 미국 ETF 종가 조회 가능 여부 확인
- 호출 제한, 과금, 서버 IP 등록 조건 확인
- 이 phase가 실패하면 구현을 시작하지 않고 provider 후보를 다시 고른다

Phase 1. 자산/부채 모델
- 현금성 자산, 투자자산, 기타 투자자산, 부채를 분리 저장
- 부채 이자와 원금상환을 현금흐름 계산에 연결
- 기존 R0 스냅샷과의 이행 경로 정의

Phase 2. 보유 종목 등록과 일일 시세 갱신
- 종목 검색 + 수량 입력
- 매일 KST 기준 시세 갱신
- Provider 실패 시 마지막 저장 가격 유지

Phase 3. 월말 자동 스냅샷
- KST 월 말일 자동 생성
- 월말 이하 마지막 거래일 종가를 사용
- 과거 스냅샷은 이후 시세 변동으로 변경하지 않음

Phase 4. 대시보드 월별 조회
- 현재 월 요약
- 과거 월 선택
- 스냅샷 기준일과 평가 기준일 분리 표기
```

## 3. 핵심 원칙

1. 별도 주식 메뉴를 만들지 않는다.

   기능 위치는 `자산/부채 > 투자자산`이다.

2. 계좌 연동을 하지 않는다.

   사용자는 증권사 로그인 정보, 계좌번호, App Key, App Secret을 입력하지 않는다.

3. 보유 종목 입력은 검색 + 수량 입력 하나로 간다.

   붙여넣기, CSV, 계좌 가져오기는 1차 범위에서 제외한다.

4. 대시보드는 조용하게 유지한다.

   종목 목록, 연금 상세, 부채 상세, 계산 설명은 상세 화면에 둔다. 대시보드는 신뢰에 필요한 요약 숫자만 보여준다.

5. 부채는 자산 신뢰도의 일부다.

   빚내서 산 자산을 자동평가하면서 관련 부채를 차감하지 않으면 FIRE 준비도가 과대평가된다.

## 4. 자산 모델

1차 릴리즈는 하나의 불투명한 `투자가능 순자산` 입력에서 벗어나야 한다.

```text
표시 순자산
= 현금성 자산
+ 투자자산
+ 부동산 자산
+ 기타 자산
- 부채
```

```text
FIRE 계산 순자산
= FIRE 계산 포함 자산
- FIRE 계산 포함 부채
```

투자자산은 아래처럼 나눈다.

```text
투자자산
= 국내 보유 종목 자동평가
+ 기타 투자자산
```

국내 보유 종목 자동평가에는 국내상장 미국 노출 ETF를 포함한다. 미국상장 증권은 글로벌 provider가 추가되기 전까지 `기타 투자자산`에 둔다.

## 5. 연금저축과 IRP

연금저축과 IRP는 계좌 성격으로 지원한다. 단, 모든 사용자에게 항상 노출하지 않는다.

기본 정책은 아래와 같다.

```text
연금저축/IRP
= 표시 순자산에는 포함
= 기본 FIRE 브릿지 계산에서는 제외
```

이유는 단순하다. FIRE 계산에는 조기은퇴 후 바로 쓸 수 있는 자산이 중요하다. 연금저축과 IRP는 실제 가계 자산이지만, 조기은퇴 직후 생활비를 바로 해결하는 브릿지 자산과는 성격이 다르다.

UI 규칙은 아래와 같다.

- 연금저축/IRP 자산이 없는 사용자에게는 기본 대시보드에 관련 카드를 보여주지 않는다.
- 등록한 사용자에게만 투자자산 상세와 계산 설명에서 조건부로 보여준다.
- 표시 순자산에 미치는 영향이 클 때만 대시보드에 짧은 설명을 보여줄 수 있다.

문구는 아래처럼 쓴다.

```text
연금저축/IRP는 표시 순자산에는 포함하고,
기본 FIRE 계산에서는 제외해요.
```

## 6. 부채 모델

부채는 1차 설계에 포함한다. 레버리지는 FIRE 숫자를 크게 왜곡할 수 있기 때문이다.

최소 부채 필드는 아래와 같다.

- 목적: 거주 부동산 관련, 투자 관련, 생활/신용 관련, 기타.
- 현재 잔액.
- 월 이자.
- 월 원금상환.
- 연결 자산: 거주 부동산, 투자자산, 연결 없음.

순자산 처리는 아래와 같다.

```text
부채 잔액
-> 표시 순자산에서 차감
-> FIRE 계산에서는 연결 자산 기준으로 앱이 기본 처리
```

부채별 기본 계산 정책은 사용자가 회계 판단을 하지 않도록 앱이 정한다.

```text
거주 부동산 관련 부채
-> 표시 순자산에서 차감
-> 기본 FIRE 계산 순자산에서는 제외
   이유: 거주 부동산도 기본 FIRE 계산에서 제외하기 때문

투자 관련 부채
-> 표시 순자산에서 차감
-> FIRE 계산 순자산에서도 차감
   이유: 투자자산을 FIRE 계산에 넣는다면 그 자산을 사기 위한 빚도 같이 반영해야 하기 때문

생활/신용 관련 부채
-> 표시 순자산에서 차감
-> FIRE 계산 순자산에서도 차감
   이유: 특정 제외 자산에 묶이지 않은 실제 상환 의무이기 때문

기타 부채
-> 표시 순자산에서 차감
-> 기본값은 FIRE 계산 순자산에서도 차감
```

사용자에게는 위 정책을 복잡하게 묻지 않는다. 입력 화면은 목적, 잔액, 이자, 원금상환만 받는다. 고급 설정에서만 계산 포함 정책을 조정할 수 있게 남긴다.

현금흐름 처리는 아래와 같다.

```text
부채 이자
-> 월 현금흐름 비용
-> 은퇴 후에도 남을 비용이면 FIRE 연지출에 포함
```

```text
원금상환
-> 월 현금흐름 지출
-> 빚 감소 효과
-> 월 자산 증가 여력에 포함
-> FIRE 연지출에서는 기본 제외
```

사용자 설명은 회계 용어 대신 아래처럼 쓴다.

```text
이자는 비용으로 보고,
원금상환은 빚이 줄어드는 효과로 계산해요.
```

대시보드 분해 표시는 아래처럼 한다.

```text
월 자산 증가 여력
= 정기투자
+ 빚 감소
+ 남은 돈
```

## 7. 보유 종목 등록 UX

1차 릴리즈의 보유 종목 등록은 하나의 흐름만 제공한다.

```text
자산/부채
-> 투자자산
-> 종목별 자동평가
-> 종목 검색
-> 수량 입력
-> 계좌 성격 선택
-> 저장
```

입력 필드는 아래와 같다.

- 검색어: 종목명 또는 종목코드.
- 선택 종목: 국내주식 또는 국내 ETF.
- 보유 수량.
- 계좌 성격: 일반, 연금저축, IRP, 기타.
- FIRE 계산 포함 정책: 계좌 성격에서 기본 도출하고 설명을 붙인다.

검색 보조 UX는 아래처럼 둔다.

- 종목명과 종목코드 검색을 모두 지원한다.
- 검색어가 넓을 때 ETF 결과를 우선 노출한다.
- 자주 쓰는 국내상장 미국 ETF 추천칩을 제공한다.
- 미국상장 티커는 이번 릴리즈의 자동평가 후보로 보여주지 않는다.

추천칩 예시는 아래와 같다.

```text
TIGER 미국S&P500
KODEX 미국나스닥100
ACE 미국배당다우존스
SOL 미국배당다우존스
```

## 8. 평가 Provider 전략

1차 provider 후보는 키움 국내 시세 API다.

Provider 경계는 작게 둔다.

```text
DomesticValuationProvider
  searchInstruments(query)
  getLastClosePrice(instrumentId, asOfDate)
```

첫 구현 후보는 아래 하나다.

```text
KiwoomDomesticValuationProvider
```

구현 전 spike에서 반드시 확인한다.

- 국내 ETF 일봉 종가 조회 가능 여부.
- 국내상장 미국 ETF 조회 가능 여부.
- 종목 검색 API 또는 종목 마스터 확보 방식.
- 호출 제한, 과금, 운영 조건.
- App Key/App Secret과 서버 IP 등록 조건.
- 주말/공휴일 월말에서 마지막 거래일 처리 방식.

키움이 필요한 국내 ETF 흐름을 안정적으로 지원하지 못하면, 구현 전에 한국투자증권 Open API 또는 별도 시장 데이터 provider를 재검토한다.

시세 갱신 정책은 아래와 같다.

```text
일일 자동 갱신
-> 매일 KST 오전에 국내 상장 종목의 마지막 거래일 종가를 갱신한다
-> 사용자가 하루라도 빨리 자산 흐름을 파악할 수 있게 한다

월말 강제 갱신
-> 월말 자동 스냅샷 생성 전에 해당 월말 이하 마지막 거래일 종가를 다시 확인한다
-> 일일 갱신 실패나 누락이 있어도 월말 스냅샷이 오래된 가격으로 고정되지 않게 한다

수동 갱신
-> 투자자산 상세 화면에서 제한된 횟수로 제공할 수 있다
-> 1차 구현에서 비용/제한 리스크가 크면 후순위로 뺀다
```

가격 오래됨 판단:

```text
현재 화면: 마지막 저장 가격을 보여주되 평가 기준일을 함께 표시
월말 스냅샷: 월말 이하 마지막 거래일 가격이 없으면 해당 종목을 자동평가에서 제외하고 경고를 남김
```

## 9. 월말 스냅샷

스냅샷은 월말 기준으로 고정한다.

```text
스냅샷 월: 2026-05
스냅샷 기준일: 2026-05-31
평가 기준일: 2026-05-31이 휴장일이면 2026-05-29
```

규칙은 아래와 같다.

- 월간 스냅샷은 해당 월의 가계 재무 사진이다.
- 보고 기준일은 월 마지막 날이다.
- 각 시세는 실제 평가 기준일을 따로 저장한다.
- 과거 월간 스냅샷은 이후 주가 변동으로 바뀌지 않는다.
- 대시보드는 과거 월별 스냅샷 조회를 지원한다.

생성 방식은 자동이다.

```text
생성 주체: 서버 스케줄러 또는 Supabase scheduled job
시간 기준: Asia/Seoul
실행 시점: 매월 마지막 날 장 마감 데이터 확보 이후
대상: 해당 월에 활성 자산/부채 입력이 있는 부부 워크스페이스
멱등성 키: couple_id + snapshot_month
```

사용자에게 월말 확정 버튼을 요구하지 않는다. 사용자는 월간 체크인을 완료하지 않아도, 입력되어 있는 최신 자산/부채 상태 기준으로 월말 자동 스냅샷을 받는다.

월중 화면은 확정 스냅샷이 아니라 현재 추정치로 구분한다.

```text
월중 대시보드
-> 현재 입력값과 마지막 저장 가격 기준의 현재 추정치

월말 스냅샷
-> 월말 기준일과 월말 이하 마지막 거래일 가격으로 고정된 기록
```

UI는 두 날짜를 구분한다.

```text
월말 기준일: 2026-05-31
주식 평가일: 마지막 거래일 2026-05-29
```

## 10. 데이터 흐름

```text
사용자가 보유 종목 등록
  -> 국내 종목 목록에서 종목 선택
  -> 수량과 계좌 성격 저장
  -> provider가 마지막 거래일 종가 조회 또는 기존 가격 사용
  -> 보유 종목 평가액 계산
  -> 투자자산 합계 계산
  -> 부채 차감
  -> FIRE 예상일 현재 추정치 재계산
  -> 매일 시세 갱신으로 현재 추정치 업데이트
  -> 월말 자동 스냅샷에서 월말 기준 기록 고정
  -> 대시보드에서 현재 추정치 또는 선택한 과거 월 표시
```

실패 흐름은 아래처럼 처리한다.

- 종목 검색 결과 없음: 종목코드로 다시 검색할 수 있는 빈 상태를 보여준다.
- Provider 장애: 마지막 저장 가격을 유지하고 오래된 가격 경고를 보여준다.
- 저장된 가격이 없음: 해당 종목은 자동평가에서 제외하고 재시도 또는 기타 투자자산 입력을 안내한다.
- 부채 입력 중 필수 잔액 누락: 조용히 0으로 처리하지 않고 저장을 막는다.

## 11. 대시보드 UX

대시보드 상단 카드는 아래 정도로 제한한다.

```text
표시 순자산
자산 - 부채

FIRE 계산 순자산
계산 포함 자산 - 계산 포함 부채

월 자산 증가 여력
정기투자 + 빚 감소 + 남은 돈
```

투자자산 요약은 아래처럼 보여준다.

```text
투자자산 ₩42,300,000
자동평가 포함 · 마지막 거래일 2026-05-29 기준
```

대시보드에서 기본 노출하지 않는다.

- 전체 보유 종목 리스트.
- 연금저축/IRP가 없는 사용자의 연금 설명.
- 미국상장 수동 입력 자산의 상세 목록.
- 증권앱처럼 보이는 성과/수익률 화면.

## 12. 보안과 개인정보

1차 릴리즈는 계좌 연동을 하지 않는 것으로 보안 범위를 줄인다.

수집하지 않는다.

- 증권사 로그인 정보.
- 증권 계좌번호.
- 사용자 소유 증권사 API 키.
- 증권사 거래내역.

수집한다.

- 사용자가 입력한 보유 종목.
- 사용자가 입력한 수량.
- 사용자가 입력한 부채 잔액과 월 상환 정보.
- provider에서 가져온 공개 시장 가격.

권한은 아래처럼 둔다.

- 부부 워크스페이스 멤버는 가계 스냅샷을 읽을 수 있다.
- 1차 릴리즈에서는 Admin이 자산/부채 입력을 관리한다.
- Lite는 조회 중심으로 둔다. Lite 입력 권한 확장은 별도 설계에서 다룬다.

## 13. 테스트 전략

순수 계산 테스트:

- 투자자산 합계는 자동평가 금액 + 기타 투자자산 금액이다.
- FIRE 계산 순자산은 포함 부채를 차감한다.
- 표시 순자산은 표시 대상 부채를 차감한다.
- 거주 부동산 관련 부채는 기본 FIRE 계산 순자산에서 제외된다.
- 투자 관련 부채와 생활/신용 관련 부채는 기본 FIRE 계산 순자산에서 차감된다.
- 연금저축/IRP는 기본적으로 표시 순자산 포함, FIRE 브릿지 계산 제외다.
- 원금상환은 월 자산 증가 여력에 포함된다.
- 이자는 월 비용에 포함된다.

Provider 테스트:

- 주말 월말에서 마지막 거래일을 선택한다.
- 일일 자동 갱신이 마지막 거래일 종가를 저장한다.
- 월말 자동 스냅샷 생성 전 월말 강제 갱신을 수행한다.
- Provider 장애 시 마지막 저장 가격을 사용한다.
- 최초 가격이 없으면 재시도 또는 수동 입력 안내 상태가 나온다.

UI 테스트:

- 검색 + 수량 입력 등록 흐름.
- 연금저축/IRP 조건부 노출.
- 부채 이자/원금상환 설명.
- 월중 대시보드는 현재 추정치로 표시된다.
- 대시보드 월 선택으로 고정된 과거 스냅샷 조회.

DB/RLS 테스트:

- 부부 워크스페이스 멤버는 스냅샷을 읽을 수 있다.
- 비멤버는 스냅샷, 보유 종목, 부채, 평가액을 읽을 수 없다.
- Admin은 1차 릴리즈의 자산/부채 입력을 변경할 수 있다.
- 월말 자동 스냅샷은 `couple_id + snapshot_month` 기준으로 중복 생성되지 않는다.
