# FIRE 커플 앱 MVP 릴리즈 설계

- 작성일: 2026-04-25
- 기준 문서: `fire_couple_app_prd_v2.md`
- 문서 목적: 기존 PRD의 방향을 유지하면서, 개발 착수 가능한 릴리즈 분할과 MVP 경계선을 정의한다.

## 1. 제품 방향

이 제품의 1차 MVP는 **부부가 월 1회 부담 없는 입력으로 같은 FIRE 숫자를 보고 돈 대화를 시작할 수 있는지**를 검증한다.

기존 PRD의 핵심 정체성은 유지한다.

- 월 1회 3분 체크인
- 역할 비대칭 UX
- 순자산, 현금흐름, FIRE 진척 관리
- 자동 연동이 아닌 월 단위 스냅샷 기반 리추얼

1차 MVP는 관계 리추얼을 중심에 두고, FIRE 계산 엔진은 작고 신뢰도 있게 포함한다.

## 2. 릴리즈 전략

PRD의 Must 범위는 버리지 않는다. 다만 모든 기능을 한 번에 만들지 않고, R0~R3로 나눠 검증한다.

### R0. Internal Alpha + Invite

목표: 리드 파트너가 혼자 신뢰 가능한 첫 FIRE 결과를 보고, 배우자 초대 링크를 만들 의향이 있는지 확인한다.

R0는 외부 MVP가 아니다. R0는 입력 신뢰도, 계산 신뢰도, 저장 흐름, 초대 의향을 검증하는 내부 알파다. 이 제품의 핵심 가설인 부부 리추얼 검증은 R0+R1을 합쳐서 판단한다.

포함 범위:

- 구글 OAuth 로그인
- 리드 파트너 온보딩
- 세후 월수입 입력
- 투자가능 순자산 입력
- 거주 부동산 순자산 입력
- 기타 순자산 입력
- 월 고정비 총액 입력
- 월 변동비 총액 입력
- 월 정기저축/투자 입력
- 기본 FIRE 결과 표시
- 기본 대시보드
- 총액 기반 월간 스냅샷 저장
- 배우자 초대 링크 생성

R0에서 제외:

- Lite 온보딩과 체크인 입력
- 고정비 상세 카테고리 CRUD
- Request
- 투자 뉴스 피드
- AI 분석
- 상세 가계부
- 고급 FIRE 설정

### R1. Couple Check-in

목표: 배우자를 초대하고, 둘 다 월 체크인을 완료해 최종 결과를 본다.

포함 범위:

- 카카오 OAuth 로그인
- Admin/Lite 역할
- 초대 링크 수락
- Lite 온보딩
- Lite 총액 3개 입력
  - 세후 월수입
  - 월 반복지출 총액
  - 월 정기저축/투자 총액
- 월 체크인 상태
- 지난달 값 재사용
- 임시 결과와 최종 결과 구분
- 부부 합산 대시보드 최소형
- 월간 리포트 최소형

### R2. Cashflow Engine

목표: 월 현금흐름을 FIRE 계산의 핵심 엔진으로 정교화한다.

포함 범위:

- 고정비 상세 카테고리
- 대출 원금/이자 분리
- 변동비 예산
- 정기저축/투자
- 실제값과 시뮬레이션값 분리
- 절감 효과 계산
- FIRE 목표 화면
- 예상 도달 시점 재계산

### R3. Ritual Loop

목표: 다음 달에도 돌아오게 만드는 반복 사용 장치를 붙인다.

포함 범위:

- Request 템플릿 4종
- 기본 알림 문구
- 월간 결과 카드
- PWA polish
- 모바일 반응형 QA
- 지인 테스트용 안정화

## 3. 쉬운 시스템 구조

이 앱은 기술적으로 복잡하게 보일 수 있지만, 제품 흐름은 네 단계로 이해한다.

```text
1. 두 사람이 숫자 확인
   ↓
2. 앱이 부부 숫자를 합산
   ↓
3. FIRE 계산
   ↓
4. 대시보드/리포트로 같이 보기
```

개발 구조도 이 흐름을 따른다.

```text
입력 화면
→ 계산기
→ 월간 스냅샷 저장
→ 결과 화면
```

핵심 개념은 월간 스냅샷이다. 확정된 월간 리포트는 그 달의 결과 사진처럼 보존한다. 나중에 현재 고정비나 자산 값을 수정해도 이미 확정된 과거 리포트가 자동으로 바뀌면 안 된다.

## 4. 도메인 경계

PRD의 폴더 구조 제안을 유지하되, 책임은 아래처럼 나눈다.

- `auth`: 로그인, 사용자 식별
- `couple`: 부부 워크스페이스, 초대, Admin/Lite 역할
- `checkin`: 월 체크인 상태, 지난달 값 재사용, 제출/확정
- `assets`: 자산/부채 월말 스냅샷
- `cashflow`: 고정비, 변동비, 대출 원금/이자, 정기저축/투자
- `fire`: 목표 자산, 예상 도달일, 절감 효과 계산
- `dashboard`: 부부 합산 결과, 임시/최종 리포트
- `requests`: Lite에게 보내는 작은 요청

대시보드는 직접 계산하지 않는다. 대시보드는 `cashflow`, `assets`, `fire`에서 만든 결과를 보여주는 역할에 집중한다.

## 5. 장기 비전과 MVP 경계선

장기적으로 이 제품은 부부 FIRE 리추얼 앱에서 출발해, 투자 관리와 가계부, 관심 섹터 정보 피드를 결합한 개인 금융 OS/SaaS로 확장한다.

다만 MVP의 검증 범위는 부부 월간 체크인과 FIRE 진척 확인에 고정한다.

### 지금 만든다

R0~R3에서 만드는 기능:

- 월 체크인
- Admin/Lite
- 자산/부채 스냅샷
- 현금흐름 + FIRE 계산
- 대시보드/월간 리포트
- Request
- PWA

### 지금 설계만 대비한다

장기 확장을 위해 데이터 구조에서만 여지를 둔다.

- 개인/부부 워크스페이스 분리 가능성
- 자산 카테고리 확장 가능성
- 투자 자산 타입 구분
- 메모/관심사/태그 연결 여지
- 데이터 내보내기 고려

### 나중에 만든다

아래 기능은 R0~R3에 포함하지 않는다.

- 상세 가계부
- 투자 뉴스 피드
- AI 분석/추천
- 포트폴리오 리밸런싱
- 마이데이터/자동 연동
- 유료 구독

경계선 원칙:

- R0~R3에 들어가는 기능은 월 1회 부부 체크인에 직접 기여해야 한다.
- 장기 비전은 기능이 아니라 확장 가능한 데이터 구조로만 반영한다.
- AI, 뉴스, 자동화는 반복 사용이 검증된 뒤 붙인다.

## 6. 예외 처리와 신뢰도 원칙

돈을 다루는 앱이므로 결과가 화려한 것보다 숫자가 왜 이렇게 나왔는지 납득되는 것이 중요하다.

### 한국형 자산 입력 원칙

R0부터 순자산을 하나의 숫자로만 받지 않는다. 한국 부부의 자산에서 거주 부동산 비중이 크고, 거주 부동산은 바로 인출 가능한 FIRE 자산이 아니기 때문이다.

R0 입력은 최소 세 가지로 나눈다.

- 투자가능 순자산: 예금, 주식, ETF, 연금저축 등 FIRE 계산에 넣을 자산
- 거주 부동산 순자산: 자가 시세에서 주택담보대출을 뺀 값. 화면에는 보여주되 R0 FIRE 계산에서는 제외
- 기타 순자산: 차량, 보증금, 애매한 자산. 화면에는 보여주되 R0 FIRE 계산에서는 제외

R0 FIRE 예상일은 투자가능 순자산만 기준으로 계산한다. 거주 부동산과 기타 순자산을 표시 순자산에는 포함할 수 있지만, 결과 화면에서 계산 제외 사실을 반드시 보여준다.

### FIRE 가정 노출

R0는 단순성을 위해 연 5% 수익률과 25배 룰을 기본값으로 사용한다. 다만 이 가정은 한국의 주거자산, 세금, 인플레이션, 연금 수령 시점, 퇴직금, 인출 순서를 모두 반영하지 않는다.

결과 화면에는 아래 성격의 문구를 반드시 노출한다.

```text
연 5%, 25배 룰 기준의 참고용 시뮬레이션이에요.
거주 부동산, 연금 수령 시점, 세금, 인플레이션은 단순화되어 있어요.
투자 자문이 아닙니다.
```

연금/퇴직금은 중요하지만 R0 계산 엔진에 넣지 않는다. 대신 R2 이후 고급 FIRE 설정 또는 은퇴 이후 현금흐름 모델에서 다룬다.

### 배우자 체크인이 끝나지 않은 경우

부분 결과는 보여준다. 다만 최종 월간 리포트는 둘 다 완료해야 확정한다.

예시 문구:

```text
현재 입력 기준 예상 도달일: 2036년 8월
배우자 체크인이 완료되면 이번 달 결과가 확정돼요.
```

### 값이 비어 있는 경우

비어 있는 값을 몰래 0으로 처리하지 않는다. 화면에는 아직 입력되지 않았음을 표시하고, 계산에 쓰는 경우 임시 계산임을 알린다.

예시 문구:

```text
Lite 반복지출이 아직 입력되지 않아 Admin 입력값 기준으로만 계산했어요.
```

### 월 자산 증가 여력이 음수인 경우

숨기지 않는다. 비난하지 않는 표현으로 보여준다.

예시 문구:

```text
이번 달 입력 기준으로 자산 증가 여력이 -32만원이에요.
고정비나 변동비를 조정하면 FIRE 예상일이 달라질 수 있어요.
```

### FIRE 예상일이 계산 불가능한 경우

월 자산 증가 여력이 0 이하이고 투자가능 순자산도 목표에 도달하지 못한 경우, 억지로 날짜를 보여주지 않는다.

예시 문구:

```text
현재 입력 기준으로는 목표 도달 시점을 계산하기 어려워요.
월 자산 증가 여력이 생기면 예상일을 보여드릴 수 있어요.
```

### 과거 리포트 보호

확정된 월간 스냅샷은 기본적으로 바뀌지 않는다. R0~R1에서는 과거 리포트를 읽기 전용으로 둔다.

### 민감정보 보호

PRD 기준대로 아래 정보는 수집하지 않는다.

- 계좌번호
- 카드번호
- 주민등록번호
- 금융기관 로그인 정보
- 상세 거래내역
- 증권 계좌번호

## 7. 테스트와 검증 기준

### R0 검증 기준

목표: 리드 파트너가 신뢰 가능한 첫 결과를 빠르게 보고, 배우자 초대 의향을 보이는가.

통과 기준:

- 리드 파트너가 온보딩을 완료할 수 있다.
- 총액 입력 후 FIRE 결과가 즉시 표시된다.
- 거주 부동산 순자산은 표시되지만 R0 FIRE 계산에서는 제외된다.
- 결과 화면에 연 5%, 25배 룰, 투자 자문 아님, 주요 단순화 가정이 표시된다.
- 새로고침해도 입력값과 결과가 유지된다.
- 배우자 초대 링크를 만들 수 있다.
- 기본 계산식 테스트가 통과한다.
- 모바일 화면에서 온보딩이 막히지 않는다.

핵심 지표:

- 리드 파트너 온보딩 완료율
- 첫 결과 도달 시간
- 첫 결과 화면 이탈률
- 초대 링크 생성률

### R1 검증 기준

목표: 배우자가 실제로 들어오는가.

통과 기준:

- 리드 파트너가 만든 초대 링크를 배우자가 열 수 있다.
- Lite가 초대 링크로 가입할 수 있다.
- Lite가 총액 3개를 입력할 수 있다.
- 한 명만 완료해도 임시 결과가 표시된다.
- 둘 다 완료하면 최종 월간 리포트가 확정된다.

핵심 지표:

- 초대 발송률
- Lite 가입률
- Lite 첫 체크인 완료율
- 부부 최종 리포트 확정률

### R2 검증 기준

목표: 고정비/현금흐름이 FIRE 계산에 가치 있게 연결되는가.

통과 기준:

- 고정비 상세 항목을 추가/수정/비활성화할 수 있다.
- 대출 원금과 이자를 분리해서 계산한다.
- 실제값과 시뮬레이션값이 섞이지 않는다.
- 절감 시뮬레이션 후 예상 도달일 변화가 표시된다.
- 계산 함수 단위 테스트가 있다.

핵심 지표:

- 고정비 상세 설정 진입률
- 절감 시뮬레이션 사용률
- 시뮬레이션 결과 저장률

### R3 검증 기준

목표: 다음 달에도 돌아오는가.

통과 기준:

- Request 4종을 보낼 수 있다.
- Lite가 Request에 응답할 수 있다.
- 지난달 값 불러오기가 정상 동작한다.
- 월간 결과 카드가 생성된다.
- PWA 설치와 모바일 사용성이 기본 수준을 만족한다.

핵심 지표:

- 2개월 연속 체크인 비율
- 월간 리포트 열람률
- Request 응답률
- PWA 재방문율

최상위 성공 기준:

```text
부부가 월 1회 같은 FIRE 숫자를 보고,
돈 대화를 조금이라도 덜 불편하게 느끼는가?
```
